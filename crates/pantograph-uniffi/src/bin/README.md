# crates/pantograph-uniffi/src/bin

UniFFI binding generation helper source boundary.

## Purpose
This directory contains checked-in helper binaries used by the UniFFI crate to
drive supported binding generation flows. It keeps generator entrypoints close
to the native binding crate without mixing helper CLI code into the public
library facade.

## Contents
| File/Folder | Description |
| ----------- | ----------- |
| `uniffi_bindgen.rs` | Helper binary entrypoint for supported UniFFI binding generation workflows. |

## Problem
Generated language bindings must match the compiled native library and metadata
for a given build. Helper entrypoints need explicit ownership so generated
artifact production is traceable rather than hidden in ad hoc shell commands.

## Constraints
- Helper binaries must not own workflow/runtime API semantics.
- Generated bindings are artifacts, not hand-maintained source.
- Generator behavior must stay compatible with the UniFFI version used by the
  crate.
- Host-language package scripts must document any additional generator tools
  they require.

## Decision
Keep generator helper binaries in this `bin/` directory and treat them as
binding-tooling adapters over the UniFFI crate. Public API changes belong in
the library wrapper first, then generator helpers and packaging scripts follow.

## Alternatives Rejected
- Keep generator commands only in developer notes: rejected because checked-in
  helper binaries are easier to review and invoke consistently.
- Put generation helper code in `lib.rs`: rejected because public FFI exports
  and developer tooling should remain separate.

## Invariants
- Helper binaries consume UniFFI metadata; they do not create independent API
  contracts.
- Generated output belongs under `target/` or package staging directories, not
  under this source directory.
- C# generation continues to use the external `uniffi-bindgen-cs` flow unless
  a supported in-crate generator path replaces it.

## Revisit Triggers
- UniFFI generator CLI behavior changes.
- C# generation moves into the checked-in helper binary.
- Binding artifacts gain a formal package manifest generated by this crate.

## Dependencies
**Internal:** parent `pantograph-uniffi` crate, UniFFI metadata, and packaging
scripts under `scripts/`.

**External:** UniFFI generator tooling and optional host-language binding
generators such as `uniffi-bindgen-cs`.

## Usage Examples
```bash
cargo run -p pantograph-uniffi --bin pantograph-uniffi-bindgen -- --help
```

## API Consumer Contract
- Inputs: compiled UniFFI crate metadata, generator arguments, and output
  directories.
- Outputs: generated binding files and generator diagnostics.
- Lifecycle: helper binaries run as developer/release tooling and exit after
  generation.
- Errors: generator failures should be surfaced directly and must not be
  swallowed by package scripts.
- Versioning: generator output must match the native library and UniFFI version
  used by the same build.

## Structured Producer Contract
- Stable fields: generated binding method names, record fields, native library
  names, and metadata-derived export sets are machine-consumed by host
  language builds.
- Defaults: output paths should default to build/package staging directories
  rather than source paths.
- Enums and labels: exported function names and enum variant names carry public
  binding semantics.
- Ordering: generated file ordering is not semantic, but generated metadata must
  match the compiled library.
- Compatibility: generated files and native libraries from different builds
  must not be mixed.
- Regeneration/migration: public UniFFI API changes must update generator
  helpers, package scripts, smoke tests, and README docs together.
