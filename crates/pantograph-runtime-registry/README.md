# pantograph-runtime-registry

Backend-owned runtime registry state machine and policy contracts for
Pantograph.

## Purpose
This crate owns runtime residency records, observation reconciliation,
reservation/admission decisions, reclaim/warmup dispositions, and technical-fit
selection contracts. The crate boundary exists so runtime policy is computed in
backend Rust instead of being rebuilt by Tauri, bindings, or frontend code.

## Contents
| File/Folder | Description |
| ----------- | ----------- |
| `Cargo.toml` | Crate manifest for the runtime-registry policy package. |

## Problem
Workflow execution needs to know which runtimes are ready, warming, busy,
stopped, unhealthy, reserved, or eligible for reclaim. If each adapter derives
that state independently, runtime admission, diagnostics, and technical-fit
selection drift. This crate provides one backend-owned state machine and
machine-readable vocabulary.

## Constraints
- The registry must not own process spawning or concrete runtime start/stop
  mechanics.
- Hosts provide observations; this crate validates and reconciles them.
- Reservation and admission outputs must be deterministic enough for tests,
  diagnostics, and future persisted consumers.
- Public DTOs must remain additive unless a coordinated contract change is
  accepted.

## Decision
Keep runtime registry policy in its own core crate. Runtime producers and app
hosts feed observations and reservation requests into this crate, then consume
typed dispositions and snapshots. Tauri and binding layers are consumers only.

## Alternatives Rejected
- Keep registry policy inside embedded runtime: rejected because multiple
  producers and host adapters need the same policy vocabulary.
- Let frontend diagnostics infer runtime state from events: rejected because
  the backend owns runtime truth and recovery decisions.
- Use raw strings for technical-fit and reclaim reasons: rejected because
  callers need stable reason codes.

## Invariants
- Runtime ids are canonicalized at the registry boundary.
- Reservation ids are generated by the registry.
- Admission, retention, reclaim, warmup, and technical-fit decisions are
  computed here, not by adapters.
- Snapshot output remains deterministic for tests and operator diagnostics.

## Revisit Triggers
- Runtime registry state must be persisted across app restarts.
- Distributed or remote runtime fleets need coordination beyond local registry
  state.
- Technical-fit selection requires external policy plugins.
- Public snapshot or reason-code shape needs a breaking change.

## Dependencies
**Internal:** `pantograph-runtime-identity`.

**External:** `serde` for DTOs and `thiserror` for typed registry errors.

## Usage Examples
```rust
use pantograph_runtime_registry::RuntimeRegistry;

let registry = RuntimeRegistry::new();
let snapshot = registry.snapshot();
```

## API Consumer Contract
- Inputs: runtime observations, reservation requests, reclaim requests, warmup
  inspections, and technical-fit requests from backend hosts.
- Outputs: typed registry snapshots, leases, dispositions, errors, and reason
  codes.
- Lifecycle: callers own concrete runtime processes; the registry owns state
  transitions and decisions over submitted facts.
- Errors: invalid transitions, missing runtimes, admission failures, and
  reservation conflicts are returned as typed Rust errors.
- Versioning: public DTO additions are compatible; renaming reason codes,
  statuses, or snapshot fields requires coordinated migration.

## Structured Producer Contract
- Stable fields: registry snapshots, reservation leases, reclaim dispositions,
  warmup dispositions, and technical-fit decisions are machine-consumed by
  runtime hosts and diagnostics.
- Defaults: absent observations mean the registry has no current fact for that
  runtime.
- Enums and labels: status and reason-code enums are semantic contracts, not
  display-only strings.
- Ordering: snapshot and eviction-candidate ordering must remain deterministic
  where callers use priority order.
- Compatibility: saved or logged diagnostics may outlive a release, so field
  additions should be additive.
- Regeneration/migration: any future schema/export generation must be updated
  together with the Rust DTOs and consumer docs.

## Testing
Run focused registry tests from the workspace root:

```bash
cargo test -p pantograph-runtime-registry
```

## Notes
- Process lifecycle belongs to runtime producers and host apps.
- This crate can say what should happen; it does not directly stop or start
  runtimes.
