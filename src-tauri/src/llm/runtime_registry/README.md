# src-tauri/src/llm/runtime_registry

## Purpose
This directory contains the app-layer `RuntimeRegistry` foundation for
Pantograph's desktop host. It exists to own runtime residency state,
reservation bookkeeping, and future admission/retention policy above the
backend-owned `InferenceGateway` facade without leaking that policy into Tauri
command handlers.

## Contents
| File/Folder | Description |
| ----------- | ----------- |
| `mod.rs` | Public registry facade, transition API, and focused state-machine tests. |
| `state.rs` | Runtime status, runtime record, and state-transition validation types. |
| `reservation.rs` | Reservation request, lease, and stored reservation-record contracts. |
| `snapshot.rs` | Machine-consumable registry snapshot types used for internal inspection and future diagnostics. |

## Problem
Pantograph needs one app-owned runtime coordinator before scheduler and
technical-fit work can proceed safely. Without a dedicated registry boundary,
residency and reservation logic would drift into gateway, workflow service, or
adapter code.

## Constraints
- `InferenceGateway` must remain the execution facade.
- This directory must stay in the app layer, not in `crates/inference`.
- Workflow service remains host-agnostic and must not depend on this module.
- The foundation must be usable before admission and eviction policy are fully
  implemented.

## Decision
Start the registry as a focused app-layer module under `src-tauri/src/llm`.
The module owns runtime records, reservation state, and snapshot generation,
while composition-root wiring in `src-tauri/src/main.rs` manages its lifetime.
Future milestones may expand policy behavior, but the ownership boundary starts
here instead of in gateway or adapters.

## Alternatives Rejected
- Add residency state directly to `InferenceGateway`.
  Rejected because gateway is the backend/infrastructure facade, not the owner
  of Pantograph scheduler policy.
- Put the registry in `pantograph-workflow-service`.
  Rejected because the workflow service must remain host-agnostic and portable
  across Pantograph hosts.

## Invariants
- Registry runtime ids and backend keys are canonicalized at the boundary.
- Reservation ids are generated by the registry, not by callers.
- Invalid state transitions are rejected rather than silently coerced.
- Snapshot output reflects registry-owned state only; it does not invent
  backend facts.

## Revisit Triggers
- Non-Tauri hosts need the same registry module and app-layer reuse outweighs
  local placement.
- The registry requires persistence or background task orchestration large
  enough to justify its own crate.
- Workflow service gains an explicit host-owned coordination extension point
  that changes the ideal placement.

## Dependencies
**Internal:** `src-tauri/src/main.rs`, `src-tauri/src/llm`, and
`pantograph-runtime-identity`.

**External:** `serde` for snapshot serialization and `thiserror` for internal
error reporting.

## Related ADRs
- `docs/adr/ADR-002-runtime-registry-ownership-and-lifecycle.md`
- Reason: ADR-002 freezes this directory as the app-layer owner of registry
  state above the gateway facade.
- Revisit trigger: a future ADR moves runtime-registry ownership into another
  Pantograph app-layer crate.

## Usage Examples
```rust
let registry = RuntimeRegistry::new();
registry.register_runtime(RuntimeRegistration::new("PyTorch", "PyTorch sidecar"));
```

## API Consumer Contract
- App composition code owns the shared `RuntimeRegistry` instance and injects it
  into host layers that need registry-backed coordination.
- Callers must register runtimes before attempting transitions or reservations.
- Transition and reservation methods return structured errors when the runtime
  is unknown or the requested state change is invalid.

## Structured Producer Contract
- Snapshot types in `snapshot.rs` are machine-consumable internal contracts for
  later diagnostics and inspection surfaces.
- Runtime ids and backend keys are emitted in canonical form.
- Reservation and runtime lists are sorted deterministically in snapshots.
- When this contract expands, new fields should be additive unless a deliberate
  migration is recorded.
