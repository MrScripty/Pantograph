## Objective

Define the staged architecture and implementation order for a run-centric
Pantograph GUI workbench where Scheduler is the default landing page and the
selected run provides context for Diagnostics, Graph, I/O Inspector, Library,
and Network pages.

## Scope

### In Scope

- Backend architecture stages required before frontend pages can render
  authoritative facts.
- Typed diagnostic event ledger architecture for scheduler, runtime, node, I/O,
  retention, and Library/Pumas observations.
- Frontend shell and page cutover stages.
- API projection boundaries between backend services and GUI consumers.
- Verification and refactor gates needed for a broad cross-layer change.

### Out of Scope

- Exact source edits before a stage implementation pass begins.
- Full Iroh peer discovery and distributed scheduling implementation.
- Full Node Lab agent implementation.
- Final visual design tokens beyond standards-aligned layout direction.
- Backwards-compatible support for existing saved workflow/run records after
  the identity and versioning cutover.

## Inputs

### Problem

The current UI is split between a drawing-to-Svelte surface and a workflow
graph view. The target system is an execution workbench where runs, queued
work, scheduler decisions, workflow versions, I/O artifacts, library assets,
and local/future network capacity are visible through coherent pages.

### Constraints

- Backend-owned facts must drive durable GUI state.
- The frontend cannot reconstruct canonical scheduler, diagnostics, retention,
  or workflow-version truth.
- Queue and run mutation authority belongs to backend services.
- Diagnostics extensibility uses a typed append-only event ledger with strict
  backend-owned writers and rebuildable projections, not free-form metadata
  writes.
- Rebuildable projections must be durable materialized read models with
  event-sequence cursors. Full replay is for migration, repair,
  projection-version changes, and tests, not normal startup or page load.
- The current app root owns a canvas/workflow toggle; the new workbench may
  replace that model through an intentional shell cutover rather than a
  compatibility layer.
- Existing source directories require README updates when ownership changes.
- SQLite-backed diagnostics and attribution history already exist and should
  be extended deliberately rather than bypassed.

### Assumptions

- The first implementation remains local-first and single-instance.
- Durable storage can continue using the existing local diagnostics/runtime
  persistence approach unless an implementation stage proves otherwise.
- Workflow identity is stable and user named.
- The GUI is allowed privileged admin/developer actions that normal client
  sessions cannot perform.
- Active-run selection is session-local frontend UI state and does not persist
  across restarts.
- Existing persisted workflows, run history, and diagnostics may be invalidated,
  deleted, or regenerated by the Stage `01` breaking contract cutover.

### Dependencies

- `pantograph-workflow-service` scheduler and workflow facade.
- `pantograph-diagnostics-ledger` durable history.
- `pantograph-runtime-attribution` client/session/bucket/run identity.
- `pantograph-node-contracts` node identity/version contracts.
- `pantograph-frontend-http-adapter` or Tauri command surfaces for GUI APIs.
- Svelte app shell, stores, and workflow graph components.

### Risks

| Risk | Impact | Mitigation |
| ---- | ------ | ---------- |
| Frontend pages are built before backend projections exist. | High | Stage backend contracts and API projections before page implementation. |
| Old identity/fingerprint fields remain partially wired after cutover. | High | Replace ambiguous contracts in Stage `01` and fail fast where old records cannot satisfy new invariants. |
| Scheduler events become logs instead of queryable facts. | High | Define event contracts and persistence/query projections before UI timelines. |
| Diagnostics flexibility becomes unvalidated arbitrary metadata. | High | Use typed event payloads, allowlisted event kinds, required schema versions, and backend-only event producers. |
| Retention deletes payloads without audit metadata. | High | Keep retention state and deletion reasons in durable records. |
| App shell migration leaves mixed old/new navigation semantics. | Medium | Make a single workbench shell the target and explicitly relocate or retire old surfaces during Stage `05`. |
| Broad work causes overlapping dirty implementation files. | Medium | Use stage gates and commit logical slices before the next stage. |

## Architecture Boundaries

### Workflow Registry And Versioning

Owns stable workflow identities, semantic versions, execution fingerprints,
presentation revisions, and immutable run snapshots.

This boundary should live in backend services/contracts, not frontend stores.

### Scheduler Control Plane

Owns queue state, estimates, delay decisions, admission, reservations, model
load/unload decisions, and scheduler events.

Only this boundary decides resource placement and execution timing.

### Typed Diagnostic Event Ledger

Own append-only typed diagnostic events for scheduler decisions, runtime/node
execution, I/O artifact observations, retention policy/version state, retention
cleanup events, local system observations, and Pumas/Library audit events.

This boundary uses a flexible durable storage shape with a strict write
contract: backend-only writers, allowlisted event kinds, typed Rust payloads,
schema versions, validation before persistence, privacy/retention classes, and
payload references for large or sensitive data.

Payload storage may be separate, but event metadata and audit history stay
queryable. Page-specific read models are durable materialized projections over
the typed event ledger. They store projection versions and event-sequence
cursors so normal operation applies only new events. Full rebuild remains
available for migration, corruption repair, projection-version changes, and
tests.

### API Projection Layer

Owns transport-safe DTOs for run lists, selected-run details, estimates,
scheduler events, workflow graph versions, I/O artifacts, Library usage, and
local Network node state.

Adapters project backend-owned facts; they must not compute policy locally.

### GUI App Shell

Owns top-level navigation, active-run UI state, page layout, and transient
interaction state.

The shell renders backend projections and sends commands back to backend APIs.

### Page Implementations

Own presentation and user workflows for Scheduler, Diagnostics, Graph, I/O
Inspector, Library, Network, and Node Lab placeholder/entry points.

## Implementation Continuity Note

The numbered stages describe product-area scope, but durable scheduler events
depend on the shared typed diagnostic event ledger. Do not implement a
scheduler-specific durable event repository as a shortcut. If Stage `02`
reaches event persistence before the shared ledger envelope, append boundary,
and validation rules exist, execute the Stage `03` ledger bootstrap first and
then continue Stage `02`.

Event ownership must also stay split:

- `run.*` owns execution lifecycle truth.
- `scheduler.*` owns scheduling decisions, estimates, queue controls,
  reservations, admission, model load/unload decisions, and authority actions.
- Page APIs consume projections that may join both families, but producers do
  not double-record the same fact in both families.
- Hot projections cover Scheduler/run-detail/current-status/active-run I/O
  page needs and stay current enough for normal rendering. Warm projections may
  catch up asynchronously or lazily and expose freshness state. Cold rebuilds
  are explicit maintenance paths.

## Public Facade Preservation Note

Use a contract-first breaking cutover for workflow identity, execution
fingerprints, workflow versions, run snapshots, and run-centric scheduler
projections. Existing ambiguous facades may be deleted or renamed once the
replacement contract for that stage is frozen and tested. The goal is a clean
architecture, not long-term compatibility with old persisted records or
overloaded API fields.

## Ownership And Lifecycle Note

Scheduler polling or push subscriptions introduced for the GUI must have one
owner per page/store, deterministic teardown, and tests proving cleanup. Prefer
event-driven updates. If temporary polling is used, scope it narrowly and
document why push/event delivery is not available yet.

## Re-Plan Triggers

- A stage needs schema changes not anticipated by its plan.
- Workflow versioning cannot replace existing public run identity contracts in
  one coherent cutover.
- Scheduler event persistence cannot fit the existing diagnostics or workflow
  service boundaries.
- Ledger-derived page reads require full event replay instead of materialized
  projection queries.
- The frontend shell migration cannot clearly relocate or retire an old surface.
- A stage touches enough files to require parallel implementation waves.

## Completion Criteria

- All numbered stages either complete or explicitly defer scoped future work.
- Scheduler opens as the default GUI page.
- Active-run context drives Diagnostics, Graph, I/O Inspector, Library, and
  Network views.
- Backend-owned version, scheduler, retention, audit, and local Network facts
  are projected to the GUI without frontend policy reconstruction.
- Required tests, README updates, and ADR updates are complete.
- Old workflow identity/fingerprint/run-history assumptions are removed from
  the active architecture or explicitly quarantined as deleted/regenerated data.
